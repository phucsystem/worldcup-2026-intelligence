# Home-Page Design Parity v2 — Summary Panel, Stakes, Results Widget

## Overview

Bring `frontend/app/page.tsx` to full visual + functional parity with the updated `prototypes/s01-brief-list.html` (650 lines of new design). Five new/enhanced sections: **Summary panel**, enhanced **Up next** (rich next-match card + fixture list with stakes), **Latest Results** widget (grouped + toggle), **What's at stake** stake-cards. *Stars to watch* + *Earlier briefs* are kept (design becomes a superset).

The two editorial pieces the API can't produce today — per-fixture **stakes** and per-group **scenario narrative lines** — are generated by the existing daily-brief LLM analyst; per-team scenario **notes** ("Through" / "Win = top" / "Out if lose") are computed deterministically by `standings_math` (hybrid). Tournament summary figures are deterministic.

Source brainstorm: `ck_plans/reports/design-to-app-parity-260620-1752-home-page-prototype-parity-report.md`

## Locked Decisions

- **Full parity**, keep Stars + Earlier (superset).
- **Hybrid stakes:** LLM writes fixture stakes + group narrative line/tag; `standings_math` computes per-team notes.
- **Stakes keyed to real `fixture_id`s** (passed into `computed_facts`) — LLM cannot invent matchups.
- **CSS strategy:** port new prototype classes into `globals.css`; components emit those class names (follows existing `result-chip`/`sparkline` precedent). Inline-Tailwind reimplementation rejected — the football-sphere `next-match`, grid `match-row`, and gradient panels are impractical inline.
- **Tournament summary** = dedicated `GET /api/tournament/summary` (small, single-responsibility, matches the discrete UI block).
- **`intelligence` persistence** = new-only (no backfill); brief re-runs daily at 7:00 AM AEST, and `/api/admin/run-brief` can repopulate on demand for testing.
- **Only `ResultsWidget` is a client component** (`"use client"` for the group toggle); all other new components stay server components.
